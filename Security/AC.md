基本元素

- 主体(subject)：访问客体的实体。一般等价于进程
  - 一般有三类主体：所有者(owner)、组(group)、世界(world)
- 客体(object)：受到控制的资源
- 访问权(access right)：主体访问客体的方式。一般有：读、写、执行、删除、创建、搜素

## 核心模型

### DAC

Discretionary Access Control, 自主访问控制

基于(用户)个人身份和访问规则控制访问。
”自主“是因为允许实体按自己意志修改或授于他人权限

使用的通常方式是访问矩阵(access matrix)：

- 横行为客体，竖列为主体；矩阵内是访问权
- 按列分解，为每个客体分出访问控制表(Access Control List, ACL)，表明用户及其访问权。ACL 可以有默认项
- 按行分解，为每个主体产生能力权证(capability ticket)，用来指定用户授权的客体和操作。

### MAC

Mandatory Access Control, 强制访问控制

通过(系统)许可和标记控制访问。“强制”是因为只有系统允许修改或授于他人权限

### RBAC

Role-Based Access Control, 基于角色的访问控制

基于用户的角色和访问规则控制访问

使用的通常方式是 NIST 的 $RBAC_{[0-3]}$

- RBAC0：用户(user)<-多对多->角色(role)；角色<-多对多->许可(permission, 也可用特权/授权/访问权)；用户激活某个角色称为会话(session)
- RBAC1：RBAC0+角色层次，使得权限可以在角色间继承
- RBAC2：RBAC0+静态安全约束。互斥角色、基数约束(最大角色用户数)、先决条件约束(高级角色需转职)、运行时约束(同一时间运行会话角色数)
- RBAC3：RBAC0+RBAC1+RBAC2+动态安全约束。会话限制(用户可拥有互斥角色，但在单次会话中不能同时激活)

### ABAC

Attribute-Based Access Control, 基于属性的访问控制

基于用户属性、资源属性和当前环境属性来控制访问。
实现较为消耗性能，通常用于 web，而非 os

## 形式化模型

### DAC, HRU

HRU(Harrison, Ruzzo, Ullman) 引入了命令(Command)的概念，每个命令由一系列基本操作组成。
HRU 通过命令的执行，描述了访问矩阵从一个状态变换到另一个状态的动态过程

基本的六种操作:

- `Enter right r into A[s,o]`: 权限变更。将权限 r 授予主体 s 对客体 o
- `Delete right r from A[s,o]`:	权限变更。从主体 s 对客体 o 的权限中删除 r
- `Create subject s`: 实体创建。在系统中创建一个新的活动实体（主体 s）
- `Create object o`: 实体创建。在系统中创建一个新的被动实体（客体 o）
-	`Destroy subject s`: 实体销毁。	从系统中移除一个已存在的主体 s
- `Destroy object o`:	实体销毁。从系统中移除一个已存在的客体 o

### 多级安全

多级安全系统(MLS, Multilevel Security)是强制访问控制在机密性和完整性上的形式化定义。
它通过为系统中的主体(用户 / 进程)、客体(文件 / 数据)均打安全密级标签，并基于安全标签的偏序关系进行强制访问控制

通常 MLS 中，机密性和完整性不相容

#### MAC, Bell-LaPadula

关心保密性/信息非授权泄露。

模型基于状态机，定义了“安全状态”的概念，并且证明了每个状态转换都是从一个安全状态转移到另一个安全状态，从而归纳证明了该系统满足了模型的安全性要求。

具有三个安全属性:

- 简单安全属性 (Simple Security Property): 不上读。主体只能读取安全级别等于或低于自身安全级别的客体
- \*属性 (star Property): 不下写。主体只能写入安全级别等于或高于自身安全级别的客体
- 自主安全属性 (Discretionary Security Property) 使用访问矩阵来规定自主访问控制。所有者可以自主授予/撤销访问权限

缺点:

- 没有提供对客体"降级"的管理: 受 BLP 模型约束的主体只能在一个安全级别读写文件，在相同或更低的级别查看文件。如果新文件合并了不同来源和级别的信息，客体密级会越来越高。(称为密级爬升 classification creep)
- 没有提供对共享客体的处理: BLP 能阻断正常的直接通信，但无法阻止两个合谋进程通过操纵系统共享资源(如文件锁、磁盘空间、CPU 时间等)来“打暗语”

#### MAC, Biba

涉及完整性/数据非授权篡改。
和 BLP 模型的结构定义一致。

具有三个安全属性:

- 简单完整性 (simple integrity): 不上写。主体只能修改完整性级别等于或低于自身完整性的客体
- 完整性紧闭 (integrity confinement): 不下读。主体只能读取完整性级别等于或高于自身完整性的客体
- 调用性质 (invocation property): 主体只能调用完整性等于或低于自身完整性的主体

缺点:

- 需要可信主体: 为了确保系统的可用性，在实现时通常需要引入“可信主体”来执行某些必要的、但可能违反规则的操作

### Clark-Wilson

涉及完整性/更侧重于商业数据环境中的事务处理逻辑，特别适合防止欺诈和人为错误。

基于两条基本概念:

1. 良构事务(will-formed transaction): 用户不能直接修改受保护的数据项，必须通过受控的程序来进行操作。这保证了所有的修改都是有记录、可审计且符合业务逻辑
2. 职责分离(separation of duty among user): 一项事务不能由单一个人从头到尾完成，必须拆解给不同角色

组件包含:

- 约束数据项(Constrained Data Item, CDI): 需要保护完整性、不能被随意修改的关键数据
- 未受约束数据项(Unconstrained Data Item, UDI): 系统外输入、尚未验证的数据
- 完整性验证过程(Integrity Verification Procedure, IVP): 用于检查 CDI 当前状态是否与外部现实一致的审计程序
- 转换过程(Transformation Procedure, TP): 唯一被允许操作 CDI 的程序或事务

核心规则由 5 条认证规则(certification rule)和 4 条执行规则(enforcement rule)共同构成。
Cr 定义了系统如何建立安全状态，而 Er 定义了系统如何维持这种安全状态:

- C1(IVP 认证): 系统的 IVP 必须保证所有 CDI 是有效的
- C2(TP 认证): 所有 TP 必须经过认证，确保它们能够正确地将 CDI 从一个有效状态转换到另一个有效状态
- E1(TP 授权列表): 系统必须维护一份列表，确保 CDI 必须仅被一个 TP 唯一操作，明确规定哪些主体有权运行哪些 TP
- E2(职责执行列表): 系统必须维护一份列表，将主体和 TP、CDI/UDI 联系起来。确保同一主体不能执行一个敏感任务的全部步骤
- C3(职责分离规则): E2 中，必须满足职责分离
- E3(身份认证): 试图执行 TP 的主体在执行前必须经过身份认证
- C4(日志记录): 所有的 TP 都必须写入只能追加的日志
- C5(UDI 转换认证): 任何将 UDI 输入并转变为 CDI 的 TP 都必须经过认证
- E4(最小特权): 只有管理员(安全官)才能修改授权列表

缺点:

- 规则爆炸: 模型要求对每一个受约束数据项和每一个转换过程进行形式化认证。在大型企业应用中，业务逻辑变更频繁，导致认证规则和维护 E1 授权列表的工作量呈指数级增长
- 无法阻止合谋: 虽然有职责分离，但无法阻止两个不同角色的人进行合谋
- 缺乏灵活性: 为了防止绕过 TP 直接修改数据，系统往往要求严禁任何超级管理员直接修改。但出现事故后很难快速设计 TP 再按照 TP 操作，执行后就破坏了 CWM 的模型假设

### Chinese Wall

侧重于利益冲突规避的商业安全模型/目的是防止信息在一个机构内部的不同客户之间非法流动，从而产生内幕交易或利益输送。

基于历史的动态访问控制模型: 一个人能看什么数据，取决于他过去看过什么数据

规则:

- 初始状态: 主体可以自由访问任何数据
- 筑墙过程: 一旦主体访问了在数据集(Data Set, DS)/利益冲突类(Conflict of Interest, CI) 中的客体 A，立即建立一道 Wall
- 阻断规则: 此后，该主体禁止访问同一个 CI 中的客体 B，但可以访问同一 DS 的 C
- 写出规则: 主体只能向墙内已拥有的对象写数据，禁止向墙外写
- 净化数据: 只有经过严格脱敏处理的、与具体 CI 无法产生有效关联的净化数据(sanitized data)可以无视写出规则/Wall

缺点:

- 时效性处理: 去年竞标的项目信息，今年是否仍构成冲突？模型本身是永久记忆的，但在现实中，法律的保密协议是有期限的。这就需要引入数据时效清洗的补充机制
- 多角色人员: 如果公司合伙人必须同时知道两个对立客户的宏观利润情况，模型会将其彻底锁死。现实中对此类人员的解决方案通常是流程隔离或合规披露
