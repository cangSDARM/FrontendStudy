加固(hardening)指: 缩小攻击面，堵住所有已知漏洞，让系统不容易被攻破、被控制的一系列动作。如
安装漏洞补丁、移出不需要的服务、配置用户组和权限、配置安全工具反病毒软件和防火墙、设置日志审计

#### Linux

- 权限组:
  - /etc/passwd / /etc/group / /etc/pam.d
  - setfacl / getfacl
- 防火墙:
  - /etc/host.allow / /etc/host.deny
  - iptables / netfilter
- 沙盒:
  - chroot

#### Windows

- 权限组:
  - 安全账户管理器SAM / SID
  - 用户账户控制UAC
  - 最高特权:
    - SeBackupPrivilege + SeRestorePrivilege 原生无视所有 NTFS ACL
    - SYSTEM 系统账户
- 加密:
  - bitlocker / EFS
- 基线测试:
  - Microsoft Baseline Security Analyzer

## 虚拟化安全

虚拟化的使用增加了需要关心的层次。

面临问题:

- 虚拟化逃脱 VM Escape: 虚拟化系统中的程序检测并侵入到真实系统
- hypervisor 漏洞: hypervisor 程序的漏洞比操作系统修复更为缓慢
- 镜像和快照管理: 访问镜像和快照会直接导致 rootkit
- 恶意软件隐身: 攻击者偷偷使用 VM，利用虚拟机无法被真实系统监测的特性绕过真机的防御

虚拟化层次:

|              方案              |                                     原理                                     |     I/O 性能      |                           代价                           |
| :----------------------------: | :--------------------------------------------------------------------------: | :---------------: | :------------------------------------------------------: |
|            全虚拟化            |                        VM-Exit → 软件模拟 → VM-Entry                         |        差         |                            \|                            |
|    半虚拟化(Virtio/Hyper-V)    |   Guest 明确知道自己在虚拟机里，装上专用驱动，和 VMM 通过共享内存协作通信    |     接近原生      |         必须在 Guest 里装东西（VMTools 省不掉）          |
|    容器/进程虚拟化(docker)     |     容器间利用linux的namespace和cgroups隔离。docker和主机用linux vm沟通      |   等同于物理机    |          Guest 必须特殊设计，支持 engine 的调度          |
| 设备直通(VFIO/PCI Passthrough) | 把物理网卡独占分配给某个虚拟机，Guest 直接和真实硬件通信，VMM 完全不介入 I/O |   等同于物理机    | 设备被独占，其他虚拟机用不了，高级功能（快照、迁移）全废 |
|             SR-IOV             |         硬件网卡自己"分身"成多个虚拟网卡，VMM 把分身直通给不同虚拟机         | 接近原生 + 可共享 |                    需要昂贵的专用硬件                    |

### Container OS

操作系统本身只作为容器的"底座"，不运行传统进程，所有应用都以容器形态存在

### Unikernel

应用在编译时，直接把内核里它需要的功能（内存管理、网络协议栈、设备驱动）静态链接进最终的二进制镜像，自带 bootloader，然后镜像直接在裸机启动

## 可信计算

Trusted Computing 指一种由工业联盟(TCG)推动的，以 TPM 为核心支撑的、旨在构建系统完整性和可信环境的宏观技术方案与标准体系。

包括:

- 硬件层: 以 TPM 为信任根
- 软件层: 可信软件栈、完整性度量架构
- 流程层: 从 BIOS 启动到操作系统加载的信任链传递

为系统完整性和可信环境提供:

- 可信启动(Authenticated Boot)
  - 启动过程中，前一个部件验证后一个部件的数字签名，验证通过后，运行后一个部件，否则就中止或复位系统
  - 一个部件启动完成后，把度量(Hash)安全的保存下来，比如扩展到 TPM 的 PCR 中
  - 后续接入 OS 后，度量会上报到 OS 的认证平台。平台会有可信白名单，若部件的版本信息不在白名单，则认为此设备是不受信任的
- 认证(Certification): 远程证明
  - 向外部第三方（如网络准入服务器）证明本平台当前运行的软件栈是可信的、未被篡改的
  - 这依赖于 TPM 内部的背书密钥 和 证明身份密钥
- 加密(Encryption):
  - TPM 负责生成和保管密钥，私钥永远不会以明文形式离开 TPM 芯片
  - 通过密封(Sealing)操作，数据只能在特定的平台配置状态下才能被解密

### TPM

可信平台模块(Trusted Platform Module)是独立的密码协处理器芯片，集成在主板上的密码运算和存储单元。
国内标准为可信密码模块(Trusted Cryptography Module，TCM)

使用证明提供可信根进行可信操作

可以有几种实现方式:

- 硬件芯片(dTPM): 单独的芯片
- 固件模拟(fTPM): CPU 内置。运行在 TEE 内
- 软件模拟(swTPM)

##### 可信根

可信根(Root of Trust)也分为三种:

- 安全存储密钥: 可信存储根(Root of Storage, RTS)
  - 存储的起点。它保管着所有不可迁移的密钥(如 EKCert)，并维护着度量值(哈希值)的 PCR
- 完整性度量: 可信度量根(Root of Measurement, RTM)
  - 测量的起点。它是系统加电后第一段执行的代码，负责去测量下一段代码的完整性，由此开启信任链
- 远程证明: 可信报告根(Root of Report，RTR)
  - 证明的起点。它能够基于 RTS 里的数据，出具一份外界无法伪造的、证明平台当前状态的报告

##### 可信操作

为操作提供支持的硬件包含:

- I/O寄存器: 用于与CPU通信，实现指令的接收与响应的发送
- 密钥算法引擎: 密码学算法的实现核心。用于生成密钥、执行加解密、数字签名等操作。可信根支持自定义密码算法扩展，目前国内生产销售和使用的 TPM2.0 均支持中国商用密码算法
- PCR(Platform Configuration Register): 平台配置寄存器，用来记录系统运行状态的寄存器，是构建度量链的关键组件。PCR 为易失性存储，可信根芯片上电后会恢复至默认值，且PCR不支持“写”操作，只支持“扩展”操作，因此 PCR 中不仅包含了各个组件内容的完整性信息，还包含了各个组件运行顺序的信息
- NV RAM(Non-volatile RAM): 非易失性存储空间，存储容量由芯片厂商决定，空间只有几百至几千字节，且该空间的读写等操作需要授权执行，因此，该空间一般用于存储敏感数据
- 授权验证模块: 规范规定芯片中的资源(如密钥、NV RAM等)需要授权使用或访问，在使用或访问资源时需要输入对应的授权信息，授权信息验证通过以后才可使用或访问这些资源
- 控制域管理: 控制域指规范中的 Hierarchy。规范规定资源按控制域管理，不同的控制域具备不同的功能，如 Lockout Hierarchy 管理与死锁相关的资源，Endorsement Hierarchy 管理与背书、身份等相关的资源
- 随机数生成器: 生成器内置了熵采集器，因此可以生成真随机序列。密钥的生成也依赖随机数生成器
- 时钟计数器: 支持抗字典攻击，依赖时钟计数器提供死锁机制
- Hash 引擎: 芯片中密码学 Hash 实现的核心，目前国内使用的 TPM2.0 支持 SM3 及 SHA256 等国际通用算法

### TEE

可信执行环境(Trusted Execution Environment)是 CPU 内部通过硬件隔离机制划出的一块安全区域(也称为“飞地”或 Enclave)。
能够保证数据在处理中的机密性与完整性——即使 OS、Hypervisor 或系统管理员本身被攻破或作恶，也无法窥探或篡改 TEE 内部正在运行的代码和明文数据

依赖三个核心机制:

- 硬件级隔离: 通过 CPU 指令集扩展(如 Intel SGX/TDX、AMD SEV、ARM TrustZone)，在物理上就将内存和计算资源的安全区域划分出来。普通操作系统完全无法访问安全世界的地址空间
- 内存加密: TEE 内部处理的数据虽为明文，但一旦数据通过总线写入外部内存，会由硬件自动加密。即便攻击者拔走内存条读取物理信号，也只能拿到密文
- 远程证明: TEE 可生成一份由 CPU 硬件签名的“报告”，向远程服务器证明: “我当前确实运行在真实的、未被篡改的 TEE 环境里，且代码版本无误”，这是建立信任的关键

### TSS

可信软件栈(Trusted Software Stack)是连接上层应用与底层 TPM 硬件的标准 API 中间件

工作流程如下:

1. 应用调用: 调用TSS接口层的创建密钥API(TSP/FAPI/ESAPI)
2. 服务层处理: TCS服务接收请求，构建TPM能识别的命令字节流
3. 驱动通信: TDDL 驱动层通过总线将字节流发送给物理 TPM 芯片
4. TPM 执行: 芯片在硬件内部完成密钥生成，返回公钥数据
5. 结果回传: 数据流原路返回给应用程序

## 云基础设施

“云”即云计算，是通过网络，弹性、按需、付费使用 IT 资源(基础设施 IaaS、平台 PaaS、软件 SaaS)的服务

四大部署模式(架构落地形态)

- 公有云: 第三方服务商提供，面向公众开放，按需付费
- 私有云: 为单一组织专属搭建，安全可控，适合政企核心业务
- 混合云: 公有云 + 私有云结合，兼顾弹性与安全，是企业主流选择
- 社区云: 多个有共同需求的组织共享的云环境（如行业联盟云）
