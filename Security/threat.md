## 分类

### 恶意软件(Malware)

- 病毒(感染)：修改正常程序使得病毒进行复制而进行感染的软件
- 蠕虫(传播)：主动寻找并感染其他机器的程序
- 木马(窃取)：表面上看起来有用的程序，但内部藏有恶意代码

四大类有效载荷

1. 数据窃取(Data Exfiltration)
2. 远程访问(Remote Access)
3. 系统破坏/扰乱(System Disruption)
4. 自主传播(Propagation)

### 网络威胁

- DoS: 通过耗尽系统资源，来减少系统的可用性
  - 网络链路攻击:
    - 直接攻击: 攻击者使用 ICMP/UDP/TCP SYN/SIP 洪泛等直接攻击目标 ip
    - 反射攻击: 攻击者伪造源 ip 为目标 ip, 发送给中继，中继反射流量给目标 ip
    - 放大攻击: 反射攻击的中继会反射过大的流量给目标 ip
  - 防范:
    - 路由过滤器: 路由验证来源 ip 是否是有效/发送地址，防止伪造
    - syn cookie: 服务器回复带握手信息 cookie 的 syn，发送后不等待；客户端回复 ack 有有效 cookie 时再重建链接
    - 修改配置: 超时窗口、连接表
    - CDN/用户验证/关闭广播请求
- 劫持
  - 点击劫持
  - DNS 劫持

### 应用与代码威胁

- 漏洞利用、代码注入
- 内存破坏、提权
- 供应链攻击
- APT 高级持续性威胁

#### UAF

Use After Free

1. 制造释放(Free): 触发漏洞，让目标对象被释放。此时，这块内存被标记为空闲，但指向它的指针（悬垂指针）仍然存在
2. 抢占内存(Occupying): 立即大量分配与释放对象大小相同或相近的新对象。堆内存管理器倾向于复用刚刚释放的内存块。如果攻击者能控制新对象的内容（比如一个虚函数表指针或函数指针），就能把伪造的数据填入这块内存
3. 触发使用(Use): 诱导程序使用那个悬垂指针。程序会认为这块内存里还是原来的对象，但实际上里面的数据已经被攻击者控制了

#### 缓冲区溢出

栈溢出

- 程序运行时，代码和全局数据以标准形式加载；因此代码总是被分配在相同的存储区域(函数地址固定)
- 函数被外部调用时，栈帧的状态(帧指针/返回地址/参数偏移)是一致的
- 缓冲区(变量)在实例化时，被置入栈帧的偏移量和长度也是固定的
- 溢出缓冲区即可导致相邻的内存区域受到破坏，进而退出程序或注入攻击函数地址

堆溢出

- 堆上分配有指针，溢出后覆写指针
- 堆上没有指针，溢出后会尝试覆写下一个堆的 header 指针

全局数据溢出

- 全局变量相邻内存区域里包含一些管理表(偏移量表、析构函数、动态加载库指针)

其他类型溢出

- 格式化字符串溢出: 某些格式化函数有不安全的格式(如 printf 的%x%p%n)，会导致内存被嗅探/写入
- 整数溢出: 让程序分配一个看似安全(实际过小)的缓冲区，或欺骗越界检查，从而触发堆/栈溢出

##### Shellcode

攻击代码称为 shellcode(汇编)

- 破坏后注入 shellcode，将函数地址设置为攻击代码起始地址，则可成功溢出攻击
  - 由于现代 ASLR(地址偏移随机化)或程序状态，注入后的代码段虽然确定，但初始地址是不确定的
  - 需要使用 NOP sled(什么也不做只填充)，保证执行
- 攻击后程序通常会崩溃，因此 shellcode 执行后也需要恢复现场，保证程序继续运行
  - 让 shellcode 另寻他处，栈上只放一个指向它的指针
  - 利用堆喷射/ROP
- shellcode 不能包含绝对地址(用相对地址)
- shellcode 不能包含数据段
  - 动态构造
  - 利用 call 指令会将返回地址压栈的特性，获取运行时地址
  - 使用 lea 配合 RIP 相对寻址
- 由于注入时 NULL 等中断字符通常会被截断，shellcode 不能包含这些
  - 使用等价的不包含 NULL 的指令
  - XOR 打散指令，运行时动态拼接回去

```c
void hello(char *tag) {
  char inp[16];
  printf("enter value for %s: ", tag);
  gets(inp);
  printf("hello your %s is %s\n", tag, inp);
}
// perl -e 'print pack("H*", "414243444546474851525354555657586162636465666768e8ffffbf948304080a4e4e4e4e0a");' | ./hello
// "hello" will execute twice and Segment Fault
```

防范:

- ASLR(地址空间布局随机化): 随机化所有关键内存区域的基址，但函数 A 和函数 B 之间的偏移不变
- DEP/NX(数据执行保护): 栈上/堆上数据不能当作程序执行。一旦 CPU 的指令指针(ip/rip)指向这片“数据”区域，硬件直接抛出异常，杀死进程
- CFI(控制流完整性): 程序在执行时，它的(控制流)应该是可以预测的。加载时即建立地址白名单
- IDS/IDS: 特征匹配拦截
- guard page: 每部分内存地址之间都有间隔。访问(注入)这些间隔将触发非法内存访问
- 带内指针认证: 将额外信息(如PAC)嵌入指针本身(利用其高位空闲位)，读取指针地址时 CPU 进行验证

##### 堆喷射

Heap Spraying, 在栈上用小字节的缓冲区溢出，把返回地址改成堆上的 shellcode 地址。
但堆的地址一般随机，因此需要在堆上申请巨量内存(喷射)

防范:

- 减少 32 位应用: 从根本上增加地址猜测的难度
- ASLR: 让 shellcode 需要申请的内存不切实际

##### ROP

Return Oriented Programming，是为了对抗 DEP 的一种技术

ROP 不在 shellcode 中插入代码，而是利用现有代码的多个有效分段(Garget)，拼接成攻击代码

而为了对抗 ASLR 等，还有 JIT-ROP。运行时分析内存地址(知道返回地址，算上偏移即可知道基址)

防范:

- Shadow Stack: 保存另外一个调用栈帧，ret 时检查两个栈帧
- Stack Canary: 函数调用时，编译器在返回地址和局部变量之间自动插入一个随机的“Canary”。函数返回前，程序检查这个值是否被改变。由于缓冲区溢出通常是连续的，要覆盖返回地址，就必然会先覆盖掉这个
