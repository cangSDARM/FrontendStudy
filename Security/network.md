## 边界安全

安全的网络经常包含 DMZ(非军事区)，保证一旦被攻破，攻击者只能拿到 DMZ 里的机器

外网 <-> DMZ: 有限开放(只开放网站、邮件等服务端口)
DMZ <-> 内网: 严格限制，之间还有一道内部防火墙。通常仅允许内网访问 DMZ，DMZ 不能进内网

通常放在 DMZ 里的设备: Web 服务器/邮件服务器/FTP 服务器/DNS 服务器/堡垒主机

### 防火墙

分类

- 包过滤防火墙: 按照包的五元组(源 IP、目的 IP、源端口、目的端口、协议)要求过滤网络包
- 状态防火墙: 按照包的状态规则过滤网络包
- 应用级网关: 按照使用包的应用白名单过滤网络包
- 电路级代理: 工作在会话层，检查会话状态过滤网络包

### 安全接入

#### 堡垒机

Bastion Host 特征

- 极致加固
  - 系统极简，只运行核心服务(如 SSH、SFTP)，关闭所有多余端口与服务
  - 不与内部网络共享认证体系，甚至需要额外任证
  - 所有访问与操作行为被全程留痕记录
- 边界定位
  - 通常部署在防火墙外侧或 DMZ，是内外网之间的桥头堡
  - 所有访问内网的流量必须经过它中转
- 单一受控网关
  - 作为唯一跳板，集中管理所有远程运维访问

#### 跳板机

Jump Server，仅提供简单跳转功能，无加固、审计、细粒度权限控制，安全性低

#### VPN

通常安全的 VPN 网络需要配合 IPSec 协议进行

## 入侵检测与防御

### 入侵检测系统 IDS

目标是发现未经授权的尝试活动并提供报警，基于“入侵者的行为和合法用户的行为制剂存在可量化的差别”假设

包含三个逻辑组件: 传感器、分析器、用户接口

可以分为

- 基于主机的 HIDS: 检测特定主机
- 基于网络的 NIDS: 检测特定网络频段
  - 基于无线网络的 WIDS: 检测特定的 AP
- 分布式/混合式

通常使用

- 异常检测 anomaly detection: 采集合法数据，定义正常行为来判断
- 特征或启发性检测 signature or heuristic detection: 收集已有非法模式，定义非法行为来判断

使用入侵检测协议 IDXP 及 入侵检测格式 IDMEF

IDS 中常包含“蜜罐”(honeypot)，包含有价值但合法用户不会访问的伪造信息，用以埋伏攻击者

### 入侵防御系统 IPS

IPS 实际上是防火墙功能的一种高级延伸，而非完全独立的物理设备。这只是一个术语问题

### 统一威胁管理 UTM

把防火墙、IDS/IPS、防病毒、反垃圾邮件、VPN、内容过滤等多个安全功能打包在一台设备上，主打「一站式、低成本、易运维」

### 流量分析与异常检测

#### Web 流量

存在几个难点

- 精准界定难题
  - 端口混淆：传统仅靠 80/443 端口识别 HTTP/HTTPS 完全不可靠，大量业务会使用非标准端口（如 8080、8443、自定义端口）提供 Web 服务，漏判会导致流量绕过代理
  - 协议隧道与伪装：攻击者可将 Web 流量封装在 SSH、VPN、ICMP 等协议中，传统包过滤无法识别
  - 多协议混合：现代 Web 依赖 WebSocket、HTTP/3（QUIC）、SSE 等新型协议，部分协议使用动态端口，难以静态匹配
  - 加密流量穿透：HTTPS 流量加密后，传统防火墙无法通过内容识别是否为 Web 流量，仅靠端口无法精准管控
- 代理部署与运维的工程化挑战
  - 性能瓶颈：所有 Web 流量经代理转发，代理成为单点性能瓶颈，高并发场景下易出现延迟、丢包，影响用户体验
  - 单点故障风险：代理宕机则全机构 Web 访问中断，需高可用集群部署，大幅提升复杂度
  - 用户体验与合规冲突：部分 Web 应用（如在线会议、实时协作、P2P 流媒体）不兼容代理，或需复杂的代理配置（如 PAC 脚本、透明代理）。
  - 用户会通过代理绕过工具（如 VPN、代理隧道、浏览器插件）规避管控，导致策略失效
  - 证书与加密适配问题：HTTPS 流量需代理做中间人（MITM）解密才能审计，需部署根证书，易引发用户隐私争议，且部分应用（如银行、金融类 App）会校验证书，导致无法正常访问
- 细粒度管控与审计的复杂性
  - 动态端口与多服务场景：机构内部 Web 服务器、第三方 SaaS 服务、云应用等使用大量端口 / 协议，难以逐一配置代理规则
  - 流量审计与合规成本：需对代理流量做日志留存、内容审计，存储与分析成本极高，且需满足等保、GDPR 等合规要求
  - 策略维护成本：业务迭代、新应用上线需持续更新代理白名单 / 规则，运维工作量随业务规模指数级增长

识别手段

|        手段        |                        原理                        |            优势             |            局限性             |
| :----------------: | :------------------------------------------------: | :-------------------------: | :---------------------------: |
| 深度包检测（DPI）  | 解析数据包 payload，匹配协议特征（如请求头、方法） |   精准识别，不受端口限制    |  加密流量需解密，性能开销大   |
| 应用识别（App-ID） |        基于协议特征、行为模式识别 Web 应用         | 可识别非标准端口的 Web 流量 |   依赖特征库，需要解密识别    |
| 端口 + 协议双校验  |         同时校验端口(80/443 等)和协议类型          |    简单高效，性能开销小     | 无法识别非标准端口的 Web 流量 |
|    流量行为分析    |  基于流量模式(如请求频率、会话特征)识别 Web 流量   |     可识别未知 Web 应用     |      误报率较高，需调优       |

落地方案

- 配置策略路由：将所有目的端口为 80/443(及自定义 Web 端口) 的流量，强制牵引至 Web 代理集群
- 终端准入控制（NAC）：仅合规终端可接入网络，未配置代理的终端无法访问互联网
- 安全访问服务边缘（SASE）架构：将 Web 代理、防火墙、CASB（云访问安全代理）等能力集成到云端
- 结合IPS/IDS：检测并阻断代理绕过工具（如 VPN、代理隧道）的流量
- 流量审计
